# ADR 0001: Выбор архитектуры микрофронтендов для Go2Asia

## Статус
Принято

## Дата решения
2025-12-10

## Контекст
Go2Asia состоит из множества функциональных модулей:
- Atlas Asia
- Pulse Asia
- Blog Asia
- Guru Nearby
- Russian Friendly Asia
- Rielt.Market
- Space / Connect
- Quest Asia

Каждый модуль развивается отдельно, имеет свою команду, UI, страницы, API и бизнес-логику.  
Одним монолитным фронтендом управлять невозможно — сложность линейно растет, сборки становятся тяжелыми, а баги в одном модуле могут ломать весь проект.

## Решение
Использовать **архитектуру микрофронтендов**, объединённых в единый **PWA Shell** на Next.js 15.

Каждый модуль:
- отдельный изолированный фронтенд-пакет,
- отдельный CI/CD,
- независимый релизный цикл,
- общий дизайн-токены и UI-компоненты,
- роутинг через единую оболочку.

## Последствия

### Плюсы
- независимость модулей,
- параллельная разработка,
- быстрые сборки,
- снижение количества конфликтов,
- возможность переписывать модули без влияния на других,
- идеальная совместимость с мультиагентной разработкой Cursor.

### Минусы
- усложнение DevOps и инфраструктуры,
- необходимость единого design system.

## Альтернативы
- монолитный SPA → отброшено,
- несколько независимых SPA без общего shell → слишком фрагментировано.
