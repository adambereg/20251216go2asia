# Blockchain Gateway Service — Roadmap

## Этап 0 — Проектирование (сейчас)

- Описание архитектуры и контрактов.
- Выделение сервиса в отдельный домен.
- Проработка интеграции с Connect, Points, NFT Service.

---

## Этап 1 — Минимально жизнеспособная реализация (MVP токеномики)

- Поддержка базовых операций:
  - `g2a_mint`,
  - `g2a_burn`,
  - `g2a_transfer`,
  - `nft_mint`.
- Простая очередь задач:
  - хранение в БД,
  - один или несколько воркеров.
- Интеграция с TonAPI или управляемым провайдером TON.
- Примитивный Listener по `tx_hash`.
- Wallet linking через подпись challenge.
- Базовые лимиты и velocity checks.

---

## Этап 2 — Усиление безопасности и устойчивости

- Перевод ключей в HSM / production‑grade Vault.
- Введение:
  - чётких runbook’ов по ротации ключей,
  - процедур восстановления.
- Расширенная антифрод‑логика:
  - более гибкие лимиты,
  - ручная блокировка пользователей/адресов.
- Расширенная метрика и мониторинг:
  - дашборды по статусам операций,
  - алерты на зависшие задачи.

---

## Этап 3 — Глубокая блокчейн‑интеграция

- Поддержка upgrade NFT (burn+mint).
- Поддержка нескольких кастодиальных кошельков (sharding по нагрузке).
- Интеграция с AML/KYT:
  - проверка адресов на «чистоту».
- Возможный переход с TonAPI на собственный full node.

---

## Этап 4 — Расширенные функции (опционально)

- Мульти‑чейн поддержка (если экосистема выйдет за пределы TON).
- Поддержка DAO‑механик:
  - голосования,
  - учёт голосов по on-chain NFT/G2A.
- Расширенные отчёты:
  - для стратегического анализа токеномики (объёмы, использования, распределения).

Основная идея: сначала запустить безопасный минимальный шлюз для TON, затем шаг за шагом усложнять его, оставаясь в рамках строгой безопасности и прозрачности.
